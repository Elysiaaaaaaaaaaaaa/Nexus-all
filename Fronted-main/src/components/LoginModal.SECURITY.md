# LoginModal 组件安全措施文档

## 概述
`LoginModal` 组件实现了与原有 `Login` 页面相同的安全防护措施，确保登录功能的安全性。

## 已实施的安全措施

### 1. 输入验证和 XSS 防护
- ✅ **输入清理**：所有用户输入都经过 `sanitizeInput()` 清理
  - 用户名：最大长度 50 字符，移除脚本标签和事件处理器
  - 邮箱：最大长度 254 字符，移除潜在恶意代码
  - 密码：不清理（由 API 层处理），但经过格式验证
- ✅ **格式验证**：使用专门的验证函数
  - `validateUsername()`：验证用户名格式（3-20字符，只允许字母数字下划线连字符）
  - `validateEmail()`：验证邮箱格式，防止邮箱注入攻击
  - `validatePassword()`：验证密码强度（最少6字符，最多128字符）
- ✅ **错误消息清理**：所有错误消息都经过 `sanitizeInput()` 清理，防止 XSS

### 2. 防暴力破解攻击
- ✅ **频率限制**：使用 `rateLimiter` 防止暴力破解
  - 最大尝试次数：5次/15分钟
  - 锁定时间：30分钟
  - 使用用户名作为 key，防止同一用户频繁尝试
- ✅ **账户锁定**：超过尝试次数后自动锁定账户
- ✅ **剩余次数提示**：显示剩余尝试次数，提醒用户

### 3. API 安全
- ✅ **使用原有 API 封装**：完全使用 `register()` 和 `login()` 函数
  - 这些函数已经包含输入验证和清理
  - 自动处理 token 存储
  - 统一的错误处理机制
- ✅ **Token 管理**：Token 由 API 层自动保存到 localStorage
- ✅ **请求拦截**：使用 `http.js` 封装，自动添加 Authorization header

### 4. 错误处理安全
- ✅ **安全日志**：使用 `safeLog()` 而不是 `console.error()`
  - 生产环境不记录详细错误信息
  - 绝不记录密码或敏感信息
  - 用户名在日志中显示为 `***`
- ✅ **错误消息**：不泄露系统内部信息
  - 不暴露 API 端点详情
  - 不暴露数据库结构
  - 不暴露堆栈跟踪（生产环境）
  - 统一错误消息格式

### 5. 数据安全
- ✅ **密码保护**：
  - 密码不记录到任何日志
  - 密码不清理（由 API 层处理加密）
  - 密码输入框使用 `type="password"` 隐藏输入
- ✅ **敏感信息**：
  - 用户名在日志中显示为 `***`
  - Token 存储在 localStorage（注意：存在 XSS 风险，但所有输入都已清理）

### 6. 用户体验安全
- ✅ **ESC 键关闭**：支持 ESC 键关闭模态框
- ✅ **防止背景滚动**：模态框打开时禁用 body 滚动
- ✅ **点击遮罩关闭**：点击遮罩层可关闭模态框
- ✅ **表单重置**：关闭模态框时自动重置所有表单数据

### 7. 代码健壮性
- ✅ **边界情况处理**：
  - 检查 `response?.user` 是否存在
  - 使用可选链操作符 (`?.`) 防止空值错误
  - 为所有可能为空的字段提供默认值
- ✅ **内存泄漏防护**：
  - 正确清理 `setInterval`
  - 正确清理事件监听器
  - 正确恢复 body 的 overflow 样式
- ✅ **React Hooks 最佳实践**：
  - 使用 `useRef` 避免 useEffect 依赖问题
  - 使用 `useCallback` 优化事件处理函数

## 安全层级

### 第一层：前端验证（LoginModal 组件）
1. 输入清理（`sanitizeInput`）
2. 格式验证（`validateUsername`, `validateEmail`, `validatePassword`）
3. 频率限制（`rateLimiter`）

### 第二层：API 层验证（api.js）
1. 参数验证
2. 输入清理
3. 请求拦截（自动添加 token）

### 第三层：HTTP 层（http.js）
1. 请求/响应拦截
2. 统一错误处理
3. Token 自动管理
4. 401 自动清除认证信息

### 第四层：后端验证（服务器端）
1. 数据库验证
2. 密码加密验证
3. Token 验证
4. 服务器端频率限制

## 注意事项

1. **localStorage 安全**：
   - Token 存储在 localStorage 中，存在 XSS 风险
   - 所有输入都经过验证和清理，降低风险
   - 建议：考虑使用 httpOnly cookie（需要后端支持）

2. **密码传输**：
   - 密码通过 HTTPS 传输（生产环境）
   - 前端不加密密码（由后端处理）
   - 密码不记录到任何日志

3. **错误信息**：
   - 401 错误统一返回"用户名或密码错误"，不泄露具体是用户名还是密码错误
   - 防止用户名枚举攻击

4. **开发模式**：
   - 开发模式可以绕过频率限制（仅用于开发测试）
   - 生产环境所有安全措施生效
